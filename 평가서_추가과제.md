# 결과물에 대한 평가 — 추가과제 (premium)


## 1. 구현 목표

필수과제(`basic/`)에서 요구사항을 정확히 구현한 뒤, "여기서 멈추지 않고 **실제 사용자가 쓰는 서비스라면 어땠을까**"를 기준으로 발전시켜 보고 싶었다. 구체적 목표는 다음과 같다.

- 4개 과제를 **하나의 디자인 언어**(색·폰트·컴포넌트)로 통일하기
- 각 과제에 **실무에서 볼 법한 기능**(실시간 검증, 검색/필터, 진행률 시각화 등)을 얹기
- 원래 요구사항(폼 검증·시맨틱·Grid/Flex·모듈·localStorage 등)은 **그대로 지키면서** 확장하기

---

## 2. 결과물에 대한 평가

디자인 토큰(`assets/theme.css`)에 색·그림자·라디우스·폰트를 `:root` 변수로 모아두고, 공유 스크립트(`assets/app.js`)에 테마 전환·토스트·스크롤 리빌을 넣어 **네 페이지가 같은 시스템을 재사용**하도록 만들었다. 그 위에 과제별로 다음을 추가했다.

| 앱 | 원래 요구사항 유지 | 추가한 창의 기능 |
|---|---|---|
| **회원가입** | 폼 검증·`fieldset/legend`·복수 체크박스·고정 헤더 | 실시간 필드 검증, **비밀번호 강도 미터**, 라이브 프로필 프리뷰, 결과 표(모달) |
| **여행지** | 시맨틱·figure/aside·video·앵커 내비 | **이미지 라이트박스**(ESC·배경 닫기), 스크롤 리빌, 스크롤스파이 내비 |
| **스토어** | `:root` 변수·Grid·Flex·반응형·다크모드 | **실시간 검색·필터·정렬**, 장바구니 카운트 배지 |
| **할 일** | 추가/완료/삭제·필터·이벤트 위임·모듈·localStorage | **SVG 진행률 도넛 링**, 하단 요약(전체·진행중·완료), 더블클릭 인라인 편집 |

목표였던 "요구사항 유지 + 실무형 확장"은 만족스럽게 달성했다고 본다. 특히 **디자인 시스템으로 묶으니 페이지마다 색·버튼이 제각각이던 문제가 사라지고 유지보수가 쉬워진 점**이 체감됐다.

---

## 3. 구현하면서 어려웠던 점

- **화려한 UI를 넣다 보니 원래 요구사항을 빠뜨릴 뻔했다.** 커스텀 UI(칩·세그먼트)로 바꾸는 과정에서 `fieldset/legend` 그룹화가 빠지거나, 고정 헤더가 `fixed`가 아니라 `sticky`로 되어 있는 등 **원본 요구를 다시 점검**해야 했다.
- **모달/라이트박스가 화면에 항상 떠 있었다.** `hidden`을 걸었는데도 오버레이가 페이지 로드 시부터 화면을 덮고 있었다.
- **진행률 링을 어떻게 그릴지** 막막했다. `<div>`로는 원형 게이지가 안 나와서 방법을 찾아야 했다.

---

## 4. 문제 해결 과정 (트러블슈팅)

### ① 모달·라이트박스가 항상 떠 있는 문제
- **문제**: 회원가입 결과 모달과 여행 라이트박스가 `hidden` 속성이 있는데도 페이지 로드 시부터 화면을 덮고 있었다.
- **원인 분석**: `hidden` 속성은 브라우저 기본 스타일에서 `display:none`으로 처리된다. 그런데 CSS에 `.result-overlay { display: grid }`, `.lightbox { display: grid }`처럼 **클래스로 `display`를 다시 지정**해 두었더니, 작성자(author) 스타일이 브라우저 기본 스타일을 이겨서 `hidden`이 무시되고 항상 보였다. (CSS 우선순위 문제)
- **해결**: `hidden`일 때를 명시적으로 다시 눌러주는 규칙을 추가했다.
  ```css
  .result-overlay[hidden] { display: none; }
  .lightbox[hidden]       { display: none; }
  ```
  `[hidden]`을 클래스와 함께 건 선택자가 더 구체적이라 우선 적용되어, 이제 JS가 `hidden`을 켜고 끌 때 정상적으로 나타났다 사라진다.
- **배운 점**: `display`를 클래스로 지정하면 **`hidden` 속성이 무력화**될 수 있다. `hidden`으로 요소를 숨기는 UI는 `[hidden]{display:none}`을 함께 걸어 안전하게 만들어야 한다. CSS는 "누가 이기는지(우선순위)"를 항상 같이 봐야 한다는 걸 체감했다.

### ② 커스텀 UI로 바꾸며 원본 요구가 빠진 문제
- **문제**: 성별·관심분야를 예쁜 칩/세그먼트로 바꾸면서 `<fieldset>`/`<legend>` 그룹화가 사라졌고, 헤더도 `position:sticky`로 되어 있었다.
- **원인 분석**: 디자인에 집중하다 보니 **가이드의 심화·세부 요구를 체크리스트로 다시 대조하지 않았기 때문**.
- **해결**: 요구사항을 한 줄씩 다시 대조해, 선택 그룹을 `<fieldset><legend>`로 감싸고 헤더를 `position:fixed`(+본문 `padding-top` 보정)로 되돌렸다.
- **배운 점**: 기능을 더할 때도 **원래 요구사항 체크리스트로 회귀 검증**을 해야 한다. 화려함이 기본기를 덮으면 안 된다.

### ③ 원형 진행률 게이지 그리기
- **문제**: 완료율을 원형 링으로 보여주고 싶은데 일반 요소로는 안 됐다.
- **원인 분석**: 원의 테두리를 부분만 채우려면 SVG `<circle>`의 둘레를 이용해야 함.
- **해결**: 둘레(`2 * Math.PI * 반지름`)를 `stroke-dasharray`로 두고, 완료율에 따라 `stroke-dashoffset`을 조절해 채워지는 링을 구현했다.
- **배운 점**: SVG의 `stroke-dashoffset`으로 "얼마나 채울지"를 수치로 제어할 수 있다는 걸 알게 됐다.

## 5. 코드에서 가장 신경 쓴 부분

- **디자인 토큰 단일 소스**: 색·폰트·그림자를 `theme.css`의 `:root`에 모아 `var()`로만 소비 → 테마·일관성·유지보수의 단일 소스.
- **관심사 분리 + 재사용**: 공유 로직(테마·토스트·리빌)은 `assets/app.js`에, 페이지 고유 로직은 각 페이지 JS에 분리.
- **요구사항 회귀 검증**: 기능을 추가한 뒤에도 가이드 필수·세부·추가 항목을 다시 대조해, 확장하면서 기본기가 빠지지 않게 함.
- **접근성 유지**: `prefers-reduced-motion` 존중, 라이트박스 ESC 닫기, `aria-label`.

## 7. 개선 및 다음 버전 계획

- **접근성**: 모달·라이트박스에 포커스 트랩(focus trap)과 `aria-modal` 포커스 관리 추가
- **성능**: 상품 검색에 디바운스(debounce) 적용(대량 데이터 대비)
- **스토어**: 장바구니를 카운트만이 아니라 담긴 상품 배열 + localStorage로 확장
- **폰트**: 현재 CDN 폰트(시스템 폴백 있음)를 로컬 번들로 바꿔 완전 오프라인 자립

### 회고
추가과제에서는 "요구사항 구현"을 넘어 **실제 서비스라면 사용자가 무엇을 필요로 할까**를 기준으로 기능을 설계했다. 그 과정에서 화려한 기능을 넣다가 기본 요구를 빠뜨릴 뻔한 것을 요구사항 체크리스트로 다시 잡아냈고, `hidden`이 무시되는 CSS 우선순위 문제처럼 **실제 브라우저에서 실행·검증하지 않았으면 못 봤을 버그**를 직접 찾아 고쳤다. 만들고 → 문제를 발견하고 → 원인을 분석하고 → 고치고 → 다시 검증하는 개발 흐름을 스스로 한 바퀴 돌려본 것이 가장 값진 경험이었다.
